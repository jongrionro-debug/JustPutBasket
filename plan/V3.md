## 2. V2에서 확인된 한계

V2는 `LLM parser + tag-first ranker` 구조로 바뀌면서
`dress가 black만 맞아서 상위에 오는 문제` 같은 기본적인 오답은 상당 부분 줄였다.

하지만 실제 실행 결과를 보면 아래 한계가 반복적으로 나타난다.

### 2-1. outfit-level tag와 item-level intent가 충돌한다

예를 들어 사용자는 `black trousers`를 찾고 있는데,
문서 데이터는 아래처럼 저장되어 있다.

- `category = top|trousers|shoes`
- `color = black`
- `detail = wide-leg white trousers|black top|...`

이 구조에서는 시스템이
`룩 전체는 black이고, trousers도 포함되어 있다`
고 판단할 수는 있어도
`바지 자체가 black인지 white인지`
를 안정적으로 판단하기 어렵다.

즉 현재 V2의 주요 문제는
`category는 item-level인데 color는 outfit-level인 경우가 많다`
는 점이다.

### 2-2. detail 문자열은 보조 힌트일 뿐 구조화된 item 정보가 아니다

현재 detail은 다음과 같이 자유 텍스트에 가깝다.

- `black and white vertical striped trousers`
- `wide-leg denim trousers with black embroidery`
- `long sleeve tunic with asymmetric hem|wide-leg white trousers|...`

이런 텍스트는 중요한 단서를 주지만,
구조화되지 않았기 때문에 rule로 해석하면 쉽게 과적합되거나 놓치는 케이스가 생긴다.

### 2-3. V2는 precision 중심이지만 semantic recall이 약하다

현재 V2는 symbolic retrieval이라
`무엇이 왜 올라왔는지` 설명하기 쉽고 제어도 쉽다.

하지만 아래 같은 경우에는 약할 수 있다.

- query 표현이 태그 vocabulary와 다를 때
- 시각적으로는 매우 유사하지만 tag가 다르게 표현된 경우
- category/color 외의 style nuance를 태그만으로 잡기 어려운 경우

즉 V2는 precision은 좋아졌지만,
large-scale semantic retrieval의 recall은 부족할 수 있다.

### 2-4. ranking이 아직 hand-tuned rule에 많이 의존한다

현재 ranker는 아래처럼 사람이 직접 정의한 규칙으로 움직인다.

- `category partial = +2`
- `color contradiction = -8`
- `detail target color contradiction = -8`

이 방식은 시작점으로는 좋지만,
케이스가 늘어날수록 규칙이 계속 누적될 수 있다.

정리하면,
V2의 한계는 아래 두 문장으로 요약된다.

1. `item-level understanding이 부족하다`
2. `semantic recall과 learned ranking이 아직 없다`

## 3. V3의 목표

V3의 목표는
V2의 `설명 가능하고 제어 가능한 tag-first 구조`를 유지하면서,
그 위에 `item-level understanding`과 `hybrid retrieval`을 추가하는 것이다.

즉 V3는 V2를 버리는 것이 아니라
V2를 아래 방향으로 확장하는 단계다.

### V3 핵심 목표

1. `outfit-level tag`에서 `item-level schema`로 확장한다
2. `tag-first precision`에 `embedding-based recall`을 더한다
3. rule만으로 결정하던 순위를 `hybrid score`로 확장한다
4. top-k 결과에 대해서는 더 정교한 rerank를 붙일 수 있게 만든다
5. relevance 평가 체계를 만들어 품질을 계속 측정할 수 있게 한다

한 줄로 요약하면:

`V3는 item-aware hybrid retrieval system이다.`

## 3-1. V3 모델 스택 결정

V3는 이번 단계에서
`Luxia + SigLIP2` 조합을 기준으로 설계한다.

이 결정은 단순한 API 교체가 아니라
각 모델의 역할을 분리해서
`text reasoning은 Luxia`,
`image-text item extraction도 Luxia`,
`multimodal retrieval은 SigLIP2`
로 고정하겠다는 의미다.

### 왜 이 조합을 쓰는가

- `Luxia`는 query를 구조화하고 text-only schema를 만드는 parser 역할에 적합하다
- `Luxia`는 bridge의 OpenAI-style chat completions를 통해 text+image 입력도 받을 수 있으므로, VLM 기반 item extraction까지 같은 계열 인터페이스로 통일할 수 있다
- `SigLIP2`는 text와 image를 같은 embedding space에 넣는 retrieval backbone 역할에 적합하다
- V3의 핵심 과제는 `item-level understanding + text-image recall`이므로, 한 모델로 전부 해결하려고 하기보다 역할을 분리하는 편이 안정적이다

### V3 기본 모델 선택

1. `query parser`

- 기본 모델: `luxia3-llm-32b-0731`
- 역할: user query를 `target_items`, `global_constraints`, `style_preferences`가 포함된 structured schema로 변환
- 선택 이유: 현재 V2 parser 흐름과 가장 자연스럽게 이어지고, text-only reasoning 단계에서는 기존 자산을 재사용하기 쉽다

2. `document item extractor`

- 기본 호출 방식: `Luxia bridge OpenAI-style VLM`
- 기본 endpoint: `https://bridge.luxiacloud.com/llm/openai/chat/completions/gpt-4o/create`
- 기본 모델: `gpt-4o-2024-08-06`
- 운영 원칙: `text_only`와 `image_assisted`는 같은 모델을 사용하고, 입력 modality만 다르게 가져간다
- 역할: `canonical_tags`, `raw_tags`, `detail`, 필요 시 image를 함께 입력으로 받아 item list를 추출
- 선택 이유: query parser와 item extractor를 모두 Luxia 계열 호출로 맞추면 운영 인터페이스가 단순해지고, 로컬 이미지는 `data:image/...;base64,...` 형태로 직접 전달할 수 있다

3. `embedding recall backbone`

- 기본 모델: `google/siglip2-base-patch16-224`
- 역할: query text와 archive image를 같은 embedding space에서 비교하는 multimodal retrieval backbone
- 선택 이유: V3의 embedding retrieval은 `text-only embedding`이 아니라 `text-image retrieval`이기 때문이다

4. `optional text embedding`

- 필요 시만 사용: 별도 text embedding model
- 역할: text-to-text semantic feature, item serialization 검색, 보조 clustering
- 주의: 이 모델은 V3의 main multimodal recall backbone으로 사용하지 않는다

5. `optional reranker`

- 기본 judge 후보: `luxia3-llm-32b-0731`
- 이미지 포함 judge 후보: `Luxia VLM judge`
- 역할: top-k 후보에 대해 pairwise relevance judgment 또는 final rerank 수행
- 원칙: MVP에서는 비활성화하고, 평가에서 개선이 확인될 때만 붙인다

### 이번 단계의 운영 원칙

- text-only parsing과 reasoning 단계는 우선 `Luxia`를 사용한다
- image understanding과 VLM 단계도 `Luxia bridge OpenAI-style`로 통일한다
- item extractor의 `text_only`와 `image_assisted`는 같은 모델을 사용한다
- embedding recall 단계는 `SigLIP2`를 유지한다
- 즉 V3의 기본 스택은 `Luxia for text reasoning and image-text item extraction + SigLIP2 for retrieval`이다

## 4. V3의 핵심 구조

V3는 아래 5단계 구조를 목표로 한다.

### Step 1. Query Parser

V3 parser는 기본적으로 `Luxia`를 유지한다.

하지만 V2의 `canonical_tags`만으로 끝나지 않고,
가능하면 query를 아래처럼 더 구조화한다.

- `global_constraints`
- `target_items`
- `style_preferences`
- `required_features`
- `preferred_features`

예:

```json
{
  "target_items": [
    {
      "category": "trousers",
      "color": "black",
      "silhouette": "relaxed"
    }
  ],
  "style_preferences": {
    "mood": ["minimal"]
  }
}
```

핵심은
`사용자가 무엇을 찾는지`를
룩 전체가 아니라 `item 중심`으로 표현하는 것이다.

권장 기본 모델:

- `luxia3-llm-32b-0731`

운영 원칙:

- 우선 Luxia 기반 structured schema를 고정한다
- parser는 ranking을 직접 하지 않고 query intent를 구조화하는 역할에 집중한다
- parser failure 시에는 V2 호환 fallback schema로 degrade 가능하게 만든다

#### Multi-item query 처리 원칙

V3에서 반드시 추가해야 하는 중요한 규칙은
`multi-item query를 item별 constraint로 분해해서 유지하는 것`이다.

예를 들어:

`white trousers with vintage black jacket`

이 query는 아래처럼 해석되어야 한다.

```json
{
  "target_items": [
    {
      "category": "trousers",
      "color": "white"
    },
    {
      "category": "jacket",
      "color": "black",
      "mood": ["vintage"]
    }
  ]
}
```

핵심은
`white`, `black`, `vintage` 같은 속성을
룩 전체에 평평하게 뿌리지 않고
각 item에 귀속시키는 것이다.

즉 V3 parser는
`attribute extraction`만 하는 것이 아니라
`attribute-to-item binding`까지 책임져야 한다.

### Step 2. Document Item Extraction

V3에서 가장 중요한 신규 단계다.

현재 archive document는 아래처럼 저장된다.

- `canonical_tags`
- `raw_tags`
- `detail`

V3에서는 여기에 아래를 추가해야 한다.

- `items`
- `item_confidence`
- `item_source`

예:

```json
{
  "items": [
    {
      "category": "jacket",
      "color": "black",
      "material": "leather"
    },
    {
      "category": "trousers",
      "color": "white",
      "silhouette": "wide-leg"
    }
  ]
}
```

이 단계가 있어야
`black jacket + white trousers`
와
`black trousers`
를 구분할 수 있다.

또한 이 단계가 있어야
`white trousers with black jacket`
에서
`white가 trousers에 붙었는지`,
`black이 jacket에 붙었는지`
를 document 쪽에서도 검증할 수 있다.

권장 기본 모델:

- 기본 추출: `Luxia bridge OpenAI-style VLM`
- 기본 모델: `gpt-4o-2024-08-06`

운영 원칙:

- batch preprocessing은 `LuxiaItemExtractor` 기준으로 돌린다
- `text_only`와 `image_assisted`를 모두 지원하되, 아카이브 태깅 기본값은 `image_assisted`로 본다
- 로컬 이미지는 공개 URL이 아니라 `data:image/...;base64,...` 형태로 request body에 넣는다
- structured output은 endpoint 기능에 의존하지 않고 prompt + parser contract로 보장한다

### Step 3. Candidate Retrieval

V3는 retrieval을 두 축으로 나눈다.

1. symbolic candidate retrieval
2. embedding candidate retrieval

즉:

- V2 ranker처럼 tag score로 후보를 찾고
- `SigLIP2` text-image embedding similarity로도 후보를 찾는다
- 두 후보군을 합쳐 최종 rerank 대상으로 만든다

이 구조의 목적은
`precision은 V2에서 가져가고 recall은 embedding에서 보강하는 것`이다.

권장 기본 모델:

- `google/siglip2-base-patch16-224`

운영 원칙:

- query text embedding과 archive image embedding은 같은 `SigLIP2` backbone으로 만든다
- 별도 text embedding model은 main recall이 아니라 text-only auxiliary feature로만 사용한다
- embedding retrieval은 main score가 아니라 candidate recall stage로 제한한다

### Step 4. Hybrid Ranker

V3의 ranking은 하나의 점수만 보지 않는다.

최종 점수는 최소 아래 조합으로 구성한다.

- `item_match_score`
- `global_tag_score`
- `embedding_similarity_score`
- `detail_consistency_score`
- `penalty_score`

초기에는 hand-tuned weighted sum으로 시작한다.

예:

```text
final_score =
0.45 * item_match_score +
0.25 * global_tag_score +
0.20 * embedding_score +
0.10 * detail_consistency_score -
penalty_score
```

이후 평가 로그가 쌓이면
learned weights나 LTR로 넘어갈 수 있다.

### Step 5. Optional Reranker

top 50~100 후보에 대해서만
더 비싼 reranker를 붙인다.

후보 reranker 예시:

- text-document cross encoder
- image-text reranker
- `Luxia VLM` 기반 pairwise judge

이 단계는 필수 MVP는 아니지만,
V3 구조에는 처음부터 연결 지점을 열어둬야 한다.

## 5. V3에서 새로 추가해야 하는 것

V3는 단순 점수 조정이 아니라
새로운 데이터 구조와 모듈이 필요하다.

### 5-1. Item-level document schema

반드시 새로 추가해야 한다.

추가 필드 예시:

- `items`
- `item_confidence`
- `item_extraction_notes`

추정 수정 대상:

- [`switch_query/v2/models.py`](/Users/nojonghyeon/Documents/GitHub/For_switch_query_v2/switch_query/v2/models.py)
- 또는 V3 전용 `switch_query/v3/models.py`

추천:

- V2를 안정적으로 유지하려면 V3는 별도 namespace로 분리하는 것이 좋다
- 즉 `switch_query/v3/` 디렉터리를 새로 만드는 방향이 안전하다

### 5-2. Item extractor

새 모듈이 필요하다.

예시:

- `switch_query/v3/item_extractor.py`

역할:

- 기존 `canonical_tags`, `raw_tags`, `detail`을 입력으로 받는다
- `Luxia bridge OpenAI-style VLM` 또는 deterministic parser를 이용해 item list를 만든다
- 각 item별 category/color/material/silhouette를 추정한다

이 모듈은 V3의 핵심이다.

권장 구현 방향:

- extractor는 `text_only`와 `image_assisted`를 모두 지원한다
- `image_assisted`에서는 로컬 이미지 파일을 `data URL`로 변환해 `image_url` 파트로 함께 보낸다
- response는 OpenAI-style `choices[0].message.content`에서 JSON 문자열을 꺼내 파싱한다
- output contract는 `items`, `item_confidence`, `item_extraction_notes`로 고정한다

### 5-3. Hybrid retriever

새 모듈이 필요하다.

예시:

- `switch_query/v3/retriever.py`

역할:

- item-aware symbolic retrieval
- embedding retrieval
- candidate union

### 5-4. Hybrid ranker

새 모듈이 필요하다.

예시:

- 현재: `switch_query/v3/ranker.py`에 item-aware symbolic ranker 구현
- 이후 확장 후보: hybrid weighted score 전용 모듈 또는 `switch_query/v3/ranker.py` 확장

역할:

- symbolic/item/embedding score를 합친다
- contradiction와 consistency penalty를 적용한다
- final top-k를 만든다

### 5-5. Evaluation set

새로운 데이터 자산이 필요하다.

예시:

- `tmp/v3_eval/queries.csv`
- `tmp/v3_eval/judgments.csv`

최소한 아래를 포함해야 한다.

- query text
- positive examples
- hard negative examples
- expected failure type

V3는 이 평가셋 없이 진행하면
또 다시 체감으로만 튜닝하게 된다.

## 6. V3에서 수정해야 하는 것

V3는 기존 V2의 일부를 재사용하지만,
중요한 부분은 확장 또는 분리해야 한다.

### 6-1. Parser 출력 schema 수정

현재 V2 parser 출력:

- `canonical_tags`
- `required_features`
- `preferred_features`

V3 parser 출력에는 아래가 추가되어야 한다.

- `target_items`
- `global_constraints`
- `style_preferences`

즉 query schema가 더 풍부해져야 한다.

그리고 이 schema는
`Luxia parser output`을 기준으로 고정하는 것이 좋다.

### 6-2. Index builder 수정

현재 index는 document 단위 feature 저장이 중심이다.

V3 index는 아래를 함께 저장해야 한다.

- document-level tags
- item-level extracted schema
- optional embedding vectors

즉 index builder가 item extraction 단계까지 포함해야 한다.

여기서 embedding vectors의 기본값은
`SigLIP2 image embedding`이다.

### 6-3. Pipeline 수정

현재 V2 pipeline:

- parser -> tag ranker -> explanation

V3 pipeline:

- parser
- item-aware symbolic retrieval
- embedding retrieval
- candidate merge
- hybrid ranking
- optional rerank
- explanation

즉 retrieval이 한 단계가 아니라
`candidate generation + final ranking`
으로 분리된다.

### 6-4. Explanation 수정

V3 explanation은 아래를 설명할 수 있어야 한다.

- 어느 item이 query target과 매칭되었는가
- 전체 룩 태그는 어떻게 반영되었는가
- embedding similarity는 얼마나 기여했는가
- 어떤 contradiction 때문에 감점되었는가

예:

- `matched_item: trousers(color=black, silhouette=relaxed)`
- `global_mood_match: minimal`
- `detail_conflict: none`
- `embedding_support: medium`

### 6-5. CLI와 출력 형식 수정

V3 CLI는 아래 모드를 지원하는 것이 좋다.

- `symbolic_only`
- `embedding_only`
- `hybrid`

그리고 결과 CSV/HTML에는 아래를 추가해야 한다.

- `matched_item`
- `item_score`
- `embedding_score`
- `hybrid_score`
- `penalties`

## 7. 구현 순서

V3는 한 번에 다 만들면 위험하다.
아래 순서로 단계적으로 진행해야 한다.

현재 진행 상태는 아래 기준으로 표시한다.

- `완료`: 코드/테스트 또는 데이터 자산까지 닫힌 상태
- `부분 완료`: 초안/실험/일부 구현은 있으나 phase를 닫기엔 아직 남은 일이 있는 상태
- `미진행`: 아직 본격 구현 전

### Phase 1. Evaluation foundation

상태: `부분 완료`

현재까지 완료:

- 대표 query 셋 초안 작성
- positive / hard negative judgment CSV 작성
- V2 top-20 candidate 결과 저장

아직 남음:

- failure taxonomy를 문서/평가 자산에서 하나로 정규화
- V3 전용 metric 계산 루틴 고정
- query coverage 확장

가장 먼저 해야 할 일:

1. 대표 query 셋 정리
2. positive / hard negative 수동 라벨링
3. failure taxonomy 정리

예시 failure taxonomy:

- outfit-level color mismatch
- item color ambiguity
- multi-item binding failure
- cross-item attribute swap
- category partial over-ranking
- detail false confirmation
- embedding miss despite visual relevance

이 단계가 먼저 있어야
이후 수정이 좋아졌는지 나빠졌는지 판단할 수 있다. -> 이건 일단 나중에 진행

### Phase 2. Item schema 설계

상태: `완료`

현재까지 완료:

- item-level document schema 확정
- query `target_items` / `global_constraints` / `style_preferences` schema 확정
- Luxia 기반 V3 parser 구현 및 검증
- multi-item attribute binding validator 구현 및 테스트 고정
- parser contract / coercion / validation tests 추가

메모:

- phase 2는 lean scope 기준으로 닫았다
- formal external JSON schema 파일까지 만든 것은 아니지만, 현재 운영 계약은 코드와 테스트로 고정됨

그 다음 해야 할 일:

1. item-level document schema 초안 확정
2. query의 `target_items` schema 초안 확정
3. item extractor 입출력 형식 확정
4. multi-item query에서 attribute binding 규칙 확정

이 단계에서는 아직 retrieval을 바꾸지 않아도 된다.
먼저 schema를 고정해야 한다.

### Phase 3. Item extractor 구현

상태: `부분 완료`

현재까지 완료:

- Luxia bridge OpenAI-style VLM 기반 item extractor 구현
- `text_only` / `image_assisted` 운영 분리
- safe resize 전송 모드 추가
- compact output prompt 적용
- 중간 append 저장 및 JSON trailing text 복구 추가
- 아카이브 subset 태깅 결과(`0~819`) 생성 및 실험 번들 생성

아직 남음:

- extractor 품질 평가 루틴 정리
- 전체 archive 안정화 실행 전략 확정
- retry / failure shard 관리까지 완전히 닫기

그 다음:

1. 기존 detail/raw_tags에서 item list 생성
2. color/category/material/silhouette를 item 단위로 추정
3. extraction confidence 저장
4. `text_only`와 `image_assisted`를 분리해서 운영하고, 기본 아카이브 태깅은 `image_assisted`로 수행

여기서 중요한 것은
`완벽한 extractor`를 만들기보다
`일관된 schema`를 만드는 것이다.

### Phase 4. Item-aware symbolic ranker 구현

상태: `완료`

메모:

- lean scope 기준으로 phase를 닫았다
- V3 item schema를 사용하는 symbolic ranker 구현 및 테스트 고정 완료

현재까지 완료:

- V3 item schema를 사용하는 standalone symbolic ranker 구현
- target item exact/partial/contradiction 판정 구현
- multi-item conjunction bonus 및 partial success 처리 구현
- cross-item attribute swap hard fail penalty 구현
- fallback match / detail consistency / global constraint / style preference 보조 점수 구현
- item assignment / explanation / score breakdown 출력 구조 구현
- ranker 테스트 케이스 추가 및 tie-breaking 동작 고정

이 단계에서 V2 ranker를 확장한다.

추가 규칙:

- target item exact match
- target item color contradiction
- target item silhouette mismatch
- multi-item conjunction bonus
- item-to-item attribute binding penalty
- outfit-level 태그는 보조 점수로만 사용

이 단계가 끝나면
`black trousers`와 `black look with white trousers`
를 훨씬 안정적으로 구분할 수 있어야 한다.

특히 multi-item query에 대해서는
아래 규칙을 명시적으로 넣는 것이 좋다.

- 각 target item은 독립적으로 match 여부를 계산한다
- `white trousers`를 만족하는 item과 `black jacket`을 만족하는 item이 모두 있을 때만 강한 가산점을 준다
- `white`가 top에 붙고 `black`이 trousers에 붙는 식의 cross-item attribute swap은 강한 penalty를 준다
- 둘 중 하나만 맞는 경우는 partial success로 보고, full match보다 확실히 낮게 점수화한다

즉 V3 ranker는
`룩 전체가 대충 query와 비슷한가`
가 아니라
`query가 요구한 item set을 각각 충족하는가`
를 평가해야 한다.

### Phase 5. Embedding candidate retrieval 추가

상태: `미진행`

메모:

- SigLIP2 backbone 방향만 정리되어 있음
- V3 candidate recall 파이프라인은 아직 구현 전

이 단계에서 embedding을 다시 넣는다.

하지만 V2처럼 main score로 쓰지 않고,
candidate recall 용도로만 쓴다.

이 단계의 기본 모델은
`google/siglip2-base-patch16-224`로 고정한다.

권장 구조:

- symbolic top 100
- embedding top 100
- union top N

### Phase 6. Hybrid ranker 구현

상태: `미진행`

메모:

- weighted sum / hybrid score 조합은 아직 구현 전

이 단계에서 최종 점수 조합을 만든다.

우선은 hand-tuned weighted sum으로 시작한다.
LTR은 그 다음 단계다.

### Phase 7. Optional reranker

상태: `미진행`

메모:

- judge / reranker는 아직 붙이지 않음

top-k 후보에 대해서만
cross encoder 또는 `Luxia judge`, 필요 시 `Luxia VLM judge`를 붙인다.

이 단계는 비용이 있으므로
평가에서 유의미한 개선이 있을 때만 활성화한다.

## 8. 성공 기준

V3가 성공했다고 판단하는 기준은 아래와 같다.

### 8-1. Retrieval quality

아래 query들에서
item-level 오답이 눈에 띄게 줄어야 한다.

- `black relaxed trousers`
- `white trousers with black jacket`
- `minimal black coat`
- `red floral dress`

즉
`outfit-level black` 때문에 잘못 올라오는 결과가
의미 있게 줄어야 한다.

또한
`white trousers with black jacket` 같은 multi-item query에서
속성이 서로 뒤바뀐 결과가 상위에 대량으로 올라오는 문제가
의미 있게 줄어야 한다.

### 8-2. Explanation quality

상위 결과를 읽었을 때
왜 올라왔는지 item 수준에서 이해 가능해야 한다.

즉 explanation이 아래를 말할 수 있어야 한다.

- query가 요구한 target item
- 실제 매칭된 item
- 모순되는 item attribute

### 8-3. Evaluation metric

최소 아래를 추적해야 한다.

- top-1 precision
- top-5 precision
- top-20 precision
- hard negative exclusion rate

가능하면 query 유형별로도 분리한다.

- category-heavy query
- color-sensitive query
- mood-heavy query
- mixed query
- multi-item binding query

## 9. 권장 구현 범위

V3는 범위를 너무 크게 잡으면 오래 끌릴 수 있다.
그래서 아래처럼 나누는 것이 좋다.

### V3 MVP

현재 상태: `부분 완료`

이미 끝난 항목:

- `Luxia` 기반 parser schema 고정
- item schema 설계
- item-aware symbolic ranker

진행 중인 항목:

- item extractor 초안 및 subset archive tagging

아직 안 끝난 항목:

- evaluation set 완전 closure
- 전체 archive 기준 item extractor 안정화

즉 embedding hybrid 전에
item-aware symbolic retrieval까지는 먼저 완성하고,
그 다음 embedding recall과 hybrid ranking으로 확장한다.

### V3.5

- `SigLIP2` 기반 embedding candidate retrieval 추가
- hybrid weighted score 추가

### V4 이상

- learned reranker
- user feedback loop
- click/save 기반 learning to rank

## 10. 최종 요약

V2는 중요한 전환이었다.

- LLM parser 도입
- tag-first retrieval 전환
- explanation 연결

하지만 실제 운영 관점에서 V2의 가장 큰 한계는
`item-level intent를 outfit-level tag로 완전히 해결할 수 없다는 점`이다.

따라서 V3의 핵심은 단순한 점수 조정이 아니라
아래 두 가지를 시스템 구조에 추가하는 것이다.

1. `item-aware schema`
2. `hybrid retrieval`

즉 V3는 아래 방향으로 진행해야 한다.

- 먼저 item schema와 evaluation set을 만들고
- 그 다음 `Luxia` parser schema와 `Luxia VLM` extractor schema를 고정하고
- 현재 여기까지는 item-aware symbolic ranker까지 구현되어 있고
- 그 다음 `SigLIP2` embedding recall과 hybrid ranking을 붙인다

한 줄로 요약하면:

`V3는 Luxia 기반 text parsing과 image-text item extraction, SigLIP2 기반 multimodal recall을 결합한 item-aware hybrid retrieval system이다.`
